# `_v86_posix` — CPython binding for the `v86:posix/process` WIT

Statically-linked CPython extension that imports the v86 POSIX-extension
capability ([`~/git/v86/wit/posix.wit`](../../../v86/wit/posix.wit),
v0.1.0). Same pattern as `_compression`, `_crypto_hash`, `_ssl`, `_xxhash`:
the WIT import shows up as a wasm import on `python-component.wasm`,
satisfied at compose time by `wac`/`composectl` plugging
`v86-component.wasm` into it.

See `../../docs/tier1-v86-integration.md` for the architectural framing
(why python-wasm + v86 compose rather than v86 hosting its own Python).

## Status

Scaffolded; **not yet wired into the build, not yet runnable**. What's in
place:

- `wit/world.wit` — local world `v86-posix-import` importing
  `v86:posix/process@0.1.0`.
- `wit/deps/v86-posix/posix.wit` — vendored interface-only copy of v86's
  `posix.wit`. The header at the top of the file notes the source commit
  (v86 `cbf77ae`) and explains why the `world v86-posix {…}` block was
  stripped (avoids needing the full wasi tree in this extension's deps).
- `gen/v86_posix_import.{c,h}` + `_component_type.o` — generated by
  `wit-bindgen-cli 0.48.1` (same version as the other extensions). To
  regenerate after the upstream WIT changes:

      cd cpython-ext/_v86_posix
      cp ~/git/v86/wit/posix.wit wit/deps/v86-posix/posix.wit
      # re-strip the world block; see the vendored file's header for why
      $EDITOR wit/deps/v86-posix/posix.wit
      wit-bindgen c --world v86-posix-import --out-dir gen wit/

- `_v86_posixmodule.c` — Python C extension. Exposes
  `_v86_posix.spawn(...)` and a `Process` type (`pid`, `signal`, `wait`,
  `try_wait`; `take_stdin/out/err` raise `NotImplementedError` — see
  below). Errors are mapped to a typed exception hierarchy:

      OSError
       ├─ SpawnError
       │   ├─ ProgramNotFoundError
       │   ├─ ExecFailedError
       │   ├─ TooManyProcessesError
       │   ├─ InvalidArgumentError
       │   └─ GuestNotReadyError
       └─ SignalError
           ├─ NoSuchProcessError
           └─ InvalidSignalError

## Verifying the C compiles without a full `make build`

If `make build` is broken for unrelated reasons (e.g. macOS host-cpython
config drift), the extension's C can still be sanity-checked directly
against the wasi-sdk clang:

    WASI_SDK="$(pwd)/deps/wasi-sdk-33.0-arm64-macos"
    CPY="$(pwd)/deps/cpython"
    "$WASI_SDK/bin/clang" --target=wasm32-wasip2 \
        -DPy_BUILD_CORE_BUILTIN -DNDEBUG -g -O3 -Wall \
        "-I$CPY/cross-build/wasm32-wasip2" \
        "-I$CPY/Include" \
        "-I$CPY/Include/internal" \
        "-I$CPY/Include/internal/mimalloc" \
        "-I$(pwd)/cpython-ext/_v86_posix/gen" \
        -c cpython-ext/_v86_posix/_v86_posixmodule.c \
        -o /tmp/_v86_posixmodule.o
    "$WASI_SDK/bin/clang" --target=wasm32-wasip2 \
        -DNDEBUG -g -O3 -Wall \
        -c cpython-ext/_v86_posix/gen/v86_posix_import.c \
        -o /tmp/v86_posix_import.o

Both should produce ~40 KB WebAssembly object files cleanly.

## What's not yet done

1. ✅ **Build wiring.** Symlinks under `deps/cpython/Modules/` and the
   `Setup.local` entries are generated by `scripts/wire-cpython-ext.sh`
   (invoked from `make build`); see commit `70935f0` and the Tier-C
   end-to-end commit `aefa267`.
2. ✅ **`Lib/subprocess.py` shim.** Routes `subprocess.Popen` / `run` /
   `check_call` / `check_output` / `communicate` through
   `_v86_posix.spawn`; mapped error tree (`SpawnError → OSError`,
   `ProgramNotFoundError → FileNotFoundError`, …). See commit
   `4a11e06`.
3. ✅ **Stdio stream wrapping.** `_v86_posix.InputStream` /
   `OutputStream` now wrap `wasi:io/streams` handles as Python
   file-like objects (`read`, `readinto`, `write`, `flush`, `close`,
   context-manager protocol, `closed` property). `Process.take_stdin`
   / `take_stdout` / `take_stderr` return them when the corresponding
   stdio was `PIPE` (else `None`). With this `subprocess.run(
   capture_output=True)`, `check_output`, and `Popen.communicate`
   work in shape — they raise the same `OSError(ENOTSUP)` as the
   no-pipe path until the real v86 impl lands, at which point they
   start returning real bytes with no caller-side change.

   This implementation lives inside `_v86_posix` for now; if another
   capability extension (SSL MemoryBIO, HTTP response bodies, …)
   needs the same wrapping, lift these types into a shared
   `_wasi_streams` extension at that point. The Python surface they
   expose is intentionally minimal (no buffering, no text mode) so
   callers can wrap them with `io.BufferedReader` / `io.TextIOWrapper`
   without conflict.

4. ✅ **Composectl plan.** `plans/python-v86.json` (sibling to
   `plans/python-browser.json`) — see commits `d702b13`, `7a7d218`,
   `67d8080`.
5. ⏸ **v86-side `spawn` impl.** v86's `posix.wit` is the contract;
   `v86-posix-stub` (in `~/git/v86/crates/v86-posix-stub`) satisfies
   the import slot today with every `spawn` returning
   `SpawnError::GuestNotReady`. Real `spawn` dispatch into a
   guest-side `execve` is a v86 repo change — drop-in swap at the
   `V86_POSIX_COMPONENT_WASM` env-var level when ready.
