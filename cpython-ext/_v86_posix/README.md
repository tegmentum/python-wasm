# `_v86_posix` — CPython binding for the `v86:posix/process` WIT

Statically-linked CPython extension that imports the v86 POSIX-extension
capability ([`~/git/v86/wit/posix.wit`](../../../v86/wit/posix.wit),
v0.1.0). Same pattern as `_compression`, `_crypto_hash`, `_ssl`, `_xxhash`:
the WIT import shows up as a wasm import on `python-component.wasm`,
satisfied at compose time by `wac`/`composectl` plugging
`v86-component.wasm` into it.

See `../../docs/tier1-v86-integration.md` for the architectural framing
(why python-wasm + v86 compose rather than v86 hosting its own Python).

## Status

Scaffolded; **not yet wired into the build, not yet runnable**. What's in
place:

- `wit/world.wit` — local world `v86-posix-import` importing
  `v86:posix/process@0.1.0`.
- `wit/deps/v86-posix/posix.wit` — vendored interface-only copy of v86's
  `posix.wit`. The header at the top of the file notes the source commit
  (v86 `cbf77ae`) and explains why the `world v86-posix {…}` block was
  stripped (avoids needing the full wasi tree in this extension's deps).
- `gen/v86_posix_import.{c,h}` + `_component_type.o` — generated by
  `wit-bindgen-cli 0.48.1` (same version as the other extensions). To
  regenerate after the upstream WIT changes:

      cd cpython-ext/_v86_posix
      cp ~/git/v86/wit/posix.wit wit/deps/v86-posix/posix.wit
      # re-strip the world block; see the vendored file's header for why
      $EDITOR wit/deps/v86-posix/posix.wit
      wit-bindgen c --world v86-posix-import --out-dir gen wit/

- `_v86_posixmodule.c` — Python C extension. Exposes
  `_v86_posix.spawn(...)` and a `Process` type (`pid`, `signal`, `wait`,
  `try_wait`; `take_stdin/out/err` raise `NotImplementedError` — see
  below). Errors are mapped to a typed exception hierarchy:

      OSError
       ├─ SpawnError
       │   ├─ ProgramNotFoundError
       │   ├─ ExecFailedError
       │   ├─ TooManyProcessesError
       │   ├─ InvalidArgumentError
       │   └─ GuestNotReadyError
       └─ SignalError
           ├─ NoSuchProcessError
           └─ InvalidSignalError

## What's not yet done

1. **Build wiring.** This extension needs to be registered the same way
   `_compression` etc. are. The existing pattern adds the source to
   `Modules/Setup.local` in the cpython submodule and links
   `gen/<name>_import_component_type.o` into `python-component.wasm`.
   Same recipe applies here; not done in the scaffolding commit because
   it depends on the cpython build harness which is bigger surface than
   the extension itself.
2. **`Lib/_v86_subprocess.py` shim.** A thin shim that routes
   `subprocess.Popen` through `_v86_posix.spawn` when running under a
   build that has this capability. Needs the build wiring above, plus
   the stdio question (next).
3. **Stdio stream wrapping.** `Process.take_stdin/out/err` returns
   `wasi:io/streams` handles in the WIT. Wrapping those as Python
   file-like objects (so the rest of CPython treats them like normal
   pipes) is a cross-extension concern — the SSL, HTTP, and any future
   capability that returns streams faces the same need. The intent is
   to land that as a separate shared utility (likely in its own
   capability extension), then come back here and replace the
   `NotImplementedError` stubs with real wrappers.
4. **Composectl plan.** `plans/python-v86.json` (sibling to
   `plans/python-browser.json`) that plugs the python-component build
   that includes this extension against `v86-component.wasm`. Becomes a
   content-addressed artifact slottable into the `native-v86` backend
   in the `tegmentum:py-offload/registry` catalog.
5. **v86-side implementation.** v86's `posix.wit` is still
   contract-only (commit `cbf77ae`). A `spawn` impl that maps to a
   guest-side `execve` is a separate v86 change.
