# Componentize python.wasm

A plan to ship the browser python interpreter as a real WebAssembly
Component, built from wasi-sdk CPython (we pick the Python version), with
capabilities composed in via composectl. Replaces the static-link build for
each capability we migrate.

**Two driving objectives:**

1. **Componentize python.wasm.** Make it a real component so it composes with
   the rest of the `tegmentum:*` capability ecosystem.
2. **Stay free of any tool's CPython version pin.** We control which Python
   version we ship; the build is `wasi-sdk + Tools/wasm/wasi build`, so we
   move it on our cadence.

Together those rule out `componentize-py` as the primary build (its bundled
CPython is fixed). We use `wasm-tools component new` to wrap our own
wasi-sdk CPython into a component, and we compose capabilities into it via
`composectl emit` (or `wac plug`).

## Target architecture

```
                            (per-capability, hand-written initially)
                            ┌─────────────────────────────────────┐
                            │  Modules/_compression.c             │
                            │  Modules/_crypto_hash.c             │   ── wit-bindgen c
                            │  Modules/_<cap>.c                   │      generated bindings
                            └──────────────────┬──────────────────┘
                                               │ static link into CPython
                                               ▼
        wasi-sdk + Tools/wasm/wasi build ─▶ python.core.wasm  (wasi-p2 core module)
                                               │
                                               │ wasm-tools component new
                                               ▼
                                  python-component.wasm
                                  ── exports wasi:cli/run
                                  ── imports tegmentum:compression-multiplexer/…
                                                tegmentum:crypto-hash-multiplexer/…
                                                tegmentum:<cap>/…
                                               │
                                               │  composectl emit (plan.json)
                                               │  + capability components from CAS
                                               ▼
                                  python.composed.wasm   ◀── single browser-downloadable artifact
                                               │
                                               │ jco transpile
                                               ▼
                                  web/public/python-component/
                                               │
                                               │ + @tegmentum/wasi-polyfill
                                               ▼
                                          (browser)
```

The "fat browser artifact" comes from **composition**, not static linking
of capability code: composectl emit produces a single composed wasm, so the
browser still downloads exactly one file.

## Phases

Each phase has a clear deliverable, acceptance criteria, fallback, and an
effort estimate (eng-days, conservative).

### Phase 0 — wrap python.wasm as a component (no capabilities)

**What.** Run `wasm-tools component new` on the existing wasi-sdk
python.wasm. Verify it runs in the existing jco + wasi-polyfill web harness.

**Why first.** Proves the shell. Risk-isolated — independent of any binding
work.

**Deliverables.**

- `make python-component` target producing `python-component.wasm`.
- A check script that loads it via jco and runs `python -c "print('ok')"`.

**Acceptance.**

- Component validates with `wasm-tools component wit python-component.wasm`
  (exports `wasi:cli/run`, imports only `wasi:*`).
- Existing web demo works unchanged when pointed at `python-component.wasm`.

**Risks / fallback.**

- Risk: CPython emits some wasi-p1 imports that need the adapter. Fallback:
  `--adapt wasi_snapshot_preview1=<adapter>` (BCA ships one).
- Risk: jco rejects something about the wrap. Fallback: trace the rejection;
  possibly need a custom WIT world for the python component.

**Effort:** 1 day.

### Phase 1 — first capability: compression

**What.** Statically include `Modules/_compression.c` in CPython that calls
into `compression-multiplexer` WIT imports (generated by `wit-bindgen c`).
Compose the thin python component with `compression-multiplexer.wasm` via
composectl. End result: a single composed wasm where
`import _compression; _compression.compress(data)` works in the browser.

**Why this capability first.** We already have working comparison
(`bindings/pyzlib`) so correctness is verifiable byte-for-byte; the
multiplexer is mature; replaces `scripts/build-zlib.sh` cleanly.

**Deliverables.**

- `wit/python-component.wit` — world `python` exporting `wasi:cli/run`,
  importing `tegmentum:compression-multiplexer/compression-dispatcher`.
- `cpython-ext/_compression/` — hand-written C extension + cached
  `wit-bindgen c` output.
- CPython build integration: `Modules/Setup.local` line + the wasi-sdk
  wrapper passes the right `-l` and include paths.
- `plans/python-browser.json` — composectl plan: root = python-component,
  binding satisfies the compression import.
- `make python.composed.wasm` target invoking `composectl emit`.
- Python module `Lib/zlib.py` (or shim) routing `zlib`'s public API to
  `_compression` (so existing `import zlib` Just Works).

**Acceptance.**

- `python -c "import zlib; print(zlib.decompress(zlib.compress(b'hello'*1000)))"`
  in jco+wasi-polyfill matches the static-zlib build byte-for-byte.
- CPython's `Lib/test/test_zlib.py` (subset that doesn't need files) passes.
- Composed wasm size delta vs the current static-zlib python.wasm documented.

**Risks / fallback.**

- Risk: CPython's existing `_zlib` conflicts. Fallback: don't try to coexist
  — disable the static `_zlib`, route `zlib.py` to `_compression`.
- Risk: wit-bindgen-c canonical-ABI calls don't link cleanly with wasi-sdk
  libc. Fallback: small per-call shim layer.
- Risk: composectl emit doesn't handle this shape. Already fixed earlier in
  this session. If it regresses: fallback to `wac plug`, revisit composectl
  integration in Phase 4.

**Effort:** 3–5 days. **This phase is the architectural proof.**

### Phase 2 — hashing: crypto-hash + non-crypto hashing

**What.** Repeat the Phase 1 pattern for:

- `crypto-hash-multiplexer` → `_crypto_hash.c`, surfaced as Python
  `hashlib.new('sha256', ...)` etc., replacing `_hashlib`'s OpenSSL static
  link.
- `hashing-multiplexer` → `_xxhash.c`, a *new* Python module exposing
  xxhash/blake3/crc32c/murmur — capabilities Python's stdlib lacks.

**Deliverables.**

- Two more `Modules/_<cap>.c` extensions, two more plan bindings.
- `Lib/_hashlib_capability.py` shim exposing the `_hashlib` public API on
  top of `_crypto_hash`.

**Acceptance.**

- `import hashlib; hashlib.sha256(b'abc').hexdigest()` returns
  `ba7816bf…` in jco.
- New `import xxhash; xxhash.xxh3(b'abc').hex()` works.
- Composed size delta within budget (see Cross-cutting).

**Risks / fallback.**

- CPython's `hashlib` assumes `_hashlib` semantics we don't match. Fallback:
  keep `_hashlib` static-linked as fallback for one release.

**Effort:** 4–6 days.

### Phase 3 — TLS / SSL ✅ DONE

The full Phase 3 plan and execution log live in
[phase-3-tls.md](phase-3-tls.md). One-line summary:

- **3a (pivot):** reuse `~/git/openssl-wasm`'s existing
  `openssl:component/tls` instead of building a new tls-wasm.
- **3b.1–3b.6:** `cpython-ext/_ssl/` ships `_ssl_capability` (C extension) +
  `ssl_capability.py` (Python shim) backed by openssl-component, with
  bundled Mozilla WebPKI roots (121 CAs).
- **3p:** `wasi-polyfill` already had `plugins/ws-gateway/` — a KSW1
  WebSocket-multiplexing TCP gateway. Bumped version pins and added an
  integration test.
- **3c:** browser wiring (`web/src/python-runner.ts` registers the
  ws-gateway via policy overrides driven by `VITE_TCP_GATEWAY_URL`) +
  `docs/browser-tls.md`.
- **3d:** static OpenSSL retired from the default build; restorable via
  `STATIC_OPENSSL=1 make build`. python.wasm 34 MB → 30 MB; composed wasm
  41 MB → 37 MB.

**Validated end-to-end.** `NETWORK=1 make test-ssl-network` runs 21
assertions including real TLS 1.3 to example.com:443 with CERT_REQUIRED,
WebPKI cert validation, expired/self-signed rejection, and
`urllib.request.urlopen('https://example.com') → 200 OK` (after
`sys.modules['ssl'] = ssl_capability`).

**Effort.** ~2 days actual vs the 14–19 day estimate. Big saves: the 3a
pivot, the polyfill already being there, and 3d as an opt-in inversion
rather than a deletion.

**v1.1 deferrals** (documented in [phase-3-tls.md](phase-3-tls.md) and
[browser-tls.md](browser-tls.md)): wrap_bio/asyncio (needs memory-BIO in
openssl-component upstream), session resumption, channel binding for
SCRAM-PLUS/HTTP-2 client auth, file-path cert loading.

### Phase 4 — composectl + Makefile + CI automation

**What.** Formalize the build pipeline.

- The composectl plan goes from `plans/python-browser.json` in-tree →
  published to the CAS the same way `bindings/pyhash` does.
- `make web` builds: python.core.wasm → python-component.wasm → composectl
  emit → jco transpile → web/public/.
- CI workflow runs the end-to-end (build + browser smoke test) on every
  change.

**Acceptance.**

- `make web` from a clean tree produces a working `web/public/` consuming
  the composed python wasm.
- CI is green.

**Risks.**

- composectl's reference host is wasmtime; in-browser execution goes through
  jco directly. Confirm composectl's emit output runs unmodified in jco; if
  not, document delta + pin jco version.

**Effort:** 2–3 days.

### Phase 5 — retire static-link paths

Split into sub-phases as the work is broken up.

#### 5.0 — static OpenSSL retired from default (Phase 3d, done)

`STATIC_OPENSSL=1` and `STATIC_ZLIB=1` are now opt-in escape hatches.
The default build:

- skips `--with-openssl`, CPython auto-disables static `_ssl` + `_hashlib`
- doesn't invoke `scripts/build-zlib.sh` (the `_compression` extension
  + `Lib/zlib.py` shim cover `zlib`)
- `scripts/build-openssl.sh` only runs when `STATIC_OPENSSL=1`

#### 5.1 — `import hashlib` defaults to the capability path ✅

**State pre-5.1.** Default build had no static `_hashlib` AND
`hashlib_capability` was opt-in (`import hashlib_capability` to monkey-
patch stdlib hashlib). Plain `import hashlib` was effectively broken on
the default build — the stdlib's `import _hashlib` failed.

**What 5.1 ships.**

- `cpython-ext/_crypto_hash/hashlib.py` — a **self-contained**
  replacement for stdlib `hashlib`, installed at `deps/cpython/Lib/hashlib.py`
  by `make install-python-shims`. Routes the public hashlib API through
  `_crypto_hash` for the 9 multiplexer-supported algorithms (md5, sha1,
  sha256, sha384, sha512, sha3_256, sha3_512, blake2b, blake2s).
- For unsupported stdlib API (sha224, sha3_224, sha3_384, shake_*,
  pbkdf2_hmac, scrypt), the shim raises `NotImplementedError` with a
  message pointing at the underlying cap gap.
- `algorithms_guaranteed` / `algorithms_available` reflect what's
  actually here — calling code that probes these sets gets the truth.
- `hashlib_capability.py` is retained as a backwards-compat re-export
  (its auto-install becomes a no-op; explicit `install()` emits a
  DeprecationWarning).

**Coverage.** The 9 algorithms cover ~95% of real-world hashlib use
(SHA-256 / SHA-512 / SHA-1 / MD5 alone are the bulk). The gaps:

| Missing surface | Workaround | Path to closure |
|---|---|---|
| sha224, sha3_224, sha3_384, shake_* | use the closest covered algo (sha256 instead of sha224) | crypto-hash-multiplexer v0.2.x extension |
| pbkdf2_hmac, scrypt | rebuild with `STATIC_OPENSSL=1` for projects that depend on these | separate `kdf` WIT contract |

#### 5.2 — `import ssl` defaults to the capability path ✅

**State pre-5.2.** `Lib/ssl_capability.py` was installed, but plain
`import ssl` was broken on the default build (no static `_ssl`; stdlib
ssl.py's `import _ssl` failed). User code had to opt in via
`sys.modules['ssl'] = ssl_capability`.

**What 5.2 ships.** A two-layer architecture:

1. `cpython-ext/_ssl/ssl_capability.py` (Phase 3b) stays as the
   **implementation** — fully tested via `make test-ssl-network` (21
   assertions including real TLS 1.3 to example.com:443).
2. `cpython-ext/_ssl/ssl.py` (new) is a **stdlib-compat wrapper**
   installed at `deps/cpython/Lib/ssl.py`. Re-exports the
   ssl_capability surface and adds the missing stdlib symbols
   (HAS_TLSv1_*, TLSVersion / Options / VerifyFlags enums,
   SSLEOFError + sibling subclasses, MemoryBIO direct re-export,
   AlertDescription, SSLErrorNumber).

The split keeps Phase 3b implementation untouched while letting the
adapter grow the compat surface incrementally. After 5.2:

  - `import ssl` works on the default build.
  - `urllib.request.urlopen('https://...')` works without
    `sys.modules['ssl'] = ssl_capability`.
  - `ssl_capability` stays importable for back-compat.

**Coverage.** Tier-1 (urllib / requests / httpx hot paths) and most
of Tier-2 (TLSVersion / Options / VerifyFlags / SSLEOFError /
SSLWantReadError etc.) are real. Tier-3 raises `NotImplementedError`
with messages pointing at the underlying gap:

| Stub | Reason |
|---|---|
| `SSLSession`, `SSLObject` | Deferred to openssl-component v1.1 (per `docs/phase-3-tls.md`) |
| `DefaultVerifyPaths`, `get_default_verify_paths` | Capability bundles WebPKI roots; no OS verify-paths |
| `DER_cert_to_PEM_cert`, `PEM_cert_to_DER_cert`, `get_server_certificate`, `cert_time_to_seconds`, `create_connection`, `RAND_add`, `RAND_status`, `get_protocol_name` | Niche helpers not yet wired through the capability |

#### 5.3 — full removal of the static escape hatches (optional)

Once 5.2 has soaked for a release with no `STATIC_OPENSSL=1` /
`STATIC_ZLIB=1` opt-ins in CI, delete:

- `scripts/build-zlib.sh`
- `scripts/build-openssl.sh`
- The Makefile `STATIC_OPENSSL` / `STATIC_ZLIB` conditional blocks
- CPython config that probes for `--with-openssl`

The capability path becomes the only path. Conservative target: skip
this phase entirely until a real consumer asks for the cleanup.

**Effort.** 5.1 ≈ 0.5 day shipped. 5.2 ≈ 2–3 days (the rewrite is
larger). 5.3 ≈ 0.5 day if it happens.

### Phase 6 — `wit-bindgen-py` generator (conditional)

**What.** Automate the per-capability `_<cap>.c` generation.

**When.** After Phase 2. If hand-writing extensions feels like copying the
same shape three times — build the generator. If they're sufficiently
distinct — defer indefinitely.

**Scope.** A Rust tool taking a WIT interface + a small config, emitting a
CPython C extension source file using wit-bindgen-c bindings underneath.
Subset to start: records, lists, options, results, enums. Resources later.

**Effort:** 8–10 days for a usable v1.

**Decision (deferred).** Reviewing the three extensions shipped through
Phases 1 and 2 (`_compression` 172 LOC, `_crypto_hash` 345 LOC, `_xxhash`
123 LOC, total 640 LOC):

- The **mechanical** parts (name→algo table, list<u8> ↔ bytes marshaling, the
  `bytes_to_list`/`list_to_bytes` helpers) are ~40 lines per file. A
  generator could emit these.
- The **substantive** parts vary materially per capability:
  - `_compression` needs result<T,E> handling with the `bool` convention I
    documented (and got wrong first time — a generator would have inherited
    whichever convention its author chose).
  - `_crypto_hash` needed the idempotency-via-buffering design decision —
    the WIT resource's `finish` consumes state and has no clone, so
    `h.digest() == h.digest()` (a hard hashlib contract) requires Python-side
    buffering. A generator wouldn't have made that call; it would have
    emitted a subtly wrong wrapper that we'd debug the same way.
  - `_xxhash` is the simplest case and the closest to "would benefit from
    generation," but it's also the smallest file.

The conclusion: substantive design dominates the boilerplate. Phase 6 stays
deferred. Revisit if a future extension feels like the fourth copy of a
truly-identical shape. The plan does not depend on it (explicit design).

## Cross-cutting concerns

### Size budget

Current static-link python.wasm is ~30M (jco transpiles to 3 modules).

- Phase 0 (no caps): same size, ±2%.
- Phase 1 (+ compression): +~500 KB.
- Phase 2 (+ hashing, replaces OpenSSL static): **net ~−2 MB**.
- Phase 3 (+ TLS capability): TBD; depends on wrapper-vs-port choice.

Goal: composed artifact ≤ 32 MB after Phase 2. Hard ceiling: 40 MB after
Phase 3.

### Performance budget

Per-call canonical-ABI overhead in composed components is small but
non-zero. Two measurements gate progress:

- `zlib.compress(1 MB)` in jco: composed within **1.5×** of static-zlib.
- `hashlib.sha256(1 MB)`: composed within **1.5×** of static `_hashlib`.

If we miss the budget materially, that's a signal — investigate whether
wac/composectl supports any inlining, or accept the cost.

### Test strategy

- **Per-extension unit tests:** Python tests in `tests/test_<cap>_extension.py`,
  runnable natively (against a stub) and in jco against the composed wasm.
- **End-to-end:** `tests/test_browser_e2e.mjs` loads the composed wasm in
  jco+wasi-polyfill, runs a Python script using each replaced/added module,
  verifies output.
- **Regression:** subset of CPython's `Lib/test/test_zlib.py` /
  `test_hashlib.py` runs against the composed artifact.

### Compatibility

Replaced Python modules (`zlib`, `hashlib`, eventually `ssl`) keep their
**public Python API** identical. Internally they call `_compression` /
`_crypto_hash` / `_ssl` (capability-backed). User code can't tell the
difference.

### Documentation

- This file, kept current.
- README updates per phase.
- Memory file entry summarizing the new build flow.

## Decision points

Three places where we deliberately reassess.

1. **End of Phase 1.** Does the pattern feel right? Is the C extension
   boilerplate reasonable? Is size/perf within budget? **Decide:** continue /
   abandon / pivot back to static-link.
2. **After Phase 2.** Is hand-writing extensions sustainable? **Decide:**
   trigger Phase 6 (wit-bindgen-py) or defer.
3. **Before Phase 3.** Is TLS-as-a-capability the right shape, or should
   `_ssl` stay statically linked indefinitely? **Decide:** based on demand.
   Easy to defer.
